網址和網域組成

一個網址是由通訊協定、主機名稱(或子網域)與網域組成

  • https://www.nttu.edu.tw/ → 網址
  • https → 通訊協定
  • www → 主機名稱，或子網域(大多用來區分網站功能。)
  • nttu.edu.tw → 網域名稱
  • nttu → 次級網域(以學校的英文名稱或代表字來做設定與命名。)
  • edu.tw → 頂級網域(像是網站的分類，政府組機關.edu或台灣網域.tw)

TCP/IP 通訊協定

目前為網際網路的基礎通訊架構，分為 4 層：

  • 第四層 - 應用層: 提供了網絡應用程序需要的各種服務。

    HTTP、HTTPS、FTP、SMTP、SSH、DNS …

  • 第三層 - 傳輸層: 負責在不同主機之間提供端到端的通信。

    TCP / UDP

  • 第二層 - 網路互連層: 負責在不同網絡之間的數據傳輸。

    IP (Internet Protocol)

  • 第一層 - 網路存取層: 負責在相鄰網絡節點之間的物理和邏輯鏈路的傳輸。

    乙太網（Ethernet）、Wi-Fi（無線局域網協議）

資料傳遞流程

1. A應用層將資料傳送給傳輸層。在傳輸層，資料被分割成較小的封包

2. A傳輸層使用 TCP 或 UDP 協定來將封包封裝成傳輸層資料段

3. A傳輸層將資料段傳遞給 A網際網路層，並指定目的地的 IP 位址。

4. 在 A網際網路層，IP 協定負責將資料段封裝成 IP 封包。IP 封包包含源 IP 位址和目的 IP 位址。

5. IP 封包通過網際網路傳輸，在路由器之間進行轉發，直到到達目的地網路 B。

6. 在目的地網路上，目的地 B 主機的網際網路層接收到 IP 封包。

7. B網際網路層根據 IP 封包中的目的 IP 位址，將封包傳遞給相應的 B傳輸層協定（TCP 或 UDP）。

8. 目的地 B傳輸層解析接收到的資料段，並將資料段重新組合成原始的資料。

9. 最後，原始的資料在目的地 B應用層進行處理和應用。

TCP(Transmission Control Protocol )

可提供一種可靠的位元組流保證資料完整、無失真並且按順序到達。

  1. 傳輸資料被分成好幾個封包傳送時，當目的端電腦接收到每一個封包，就會傳回一個確認值
  2. 對資料的掌控上及偵錯方面，就遠勝於 UDP，但速度則不如 UDP。

TCP 三次握手(hree-way handshake)

  1. 建立連接(Browser：安安，在嗎？)
  2. 伺服器回應(Server 收到後回覆：在呀)
  3. 完成握手(Browser 收到後回覆：收到，太好了)

UDP(User Datagram Protocol)

不檢查封包是否已經到達目的地，並且不保證它們按順序到達。

  1. 只負則送出及接收，不會收到一個封包就回傳一個確認的訊息，檢查則是靠著總和檢測值來判定。
  2. 典型性應用：串流媒體（音訊和視訊等）這樣按時到達比可靠性更重要的應用

UDP 傳輸原理

傳輸方（Server）不需確認接收方（Client）是否有收到封包

DNS(Domain Name System)

用來統整 IP 與網址對應的系統，像是網際網路的電話簿

  1. 用來做為 IP 與網址的轉換
  2. 透過 DNS Server 的查詢才能讓網站的瀏覽者能正確的查看到他想看的網站

DNS 運作原理

當你在瀏覽器輸入一串網址，瀏覽器會將這串網址送到離他最近的 DNS Server 去做查詢辨識，當他找到對應的 IP 時，就會回傳這組 IP 的網站資料給你，而若沒有查詢到，則會發生像是 DNS NOT FOUND，網頁或服務無法開啓的情況。

HTTP POST

  1. 需要執行一些動作時，會傳送 Post 的 request
  2. 獲取「 指定的 」資訊，放在 request body ( Form data ) 裡面

其他的 Http Request Method

Put : 取代掉整個 request
Patch : 修改部分 request （ 較推薦 ）
Delete : 刪除資源
Head: 只要獲取 request 的 header，不要 body
Option : 可以暸解 server 提供哪些溝通方法

HTTP Code

  • 1xx: 再等等(Server 成功接收、但 Client 還要進行一些處理)
  • 2xx: 成功了
    • 200 OK：連線成功
  • 3xx: 去其他地方(重新導向)
  • 4xx: 你挫賽了（Client 端錯誤）
    • 400 Bad Request：請求語法錯誤、或資源太大…等等
    • 401 Unauthorized：未認證，可能需要登入或 Token
    • 403 Forbidden：沒有權限
    • 404 Not Found：找不到資源
  • 5xx: 我挫賽了（Server 端錯誤）
    • 500 Internal Server Error：伺服器出錯，搶票時很可能發生

SQL injection

利用網頁程式設計者忽略檢查使用者輸入內容造成攻擊

甚麼是 SQL

就是資料庫當中用來查詢資料的語言，所謂資料庫就是存放資料的地方。

資料庫的結構

  • 資料庫名稱(database) > 資料表名稱(table) > 資料欄位名稱(column) > 資料(data)
  • SELECT * FROM detail

常見的 SQL injection 類型

查詢隱藏的數據

修改 SQL 語法來查詢資料庫的其他資料。

  • 假設一個購物商場透過 URL 對伺服器請求，查詢類別為Car
    https://nttu.shop.tw/products?category=Car
  • 參數 Car 的內容會傳到後端並進行 SQL 語法的查詢
    SELECT * FROM products WHERE category = 'Car' AND released = 1
  • 此 SQL 語法的內容為
    1. 選擇所有欄位名稱 2.從資料表 products
    2. category 欄位為 Car
    3. 且 released 發布狀態為 1 （已發布）
  • 我們可以嘗試將 AND released = 1 透過 -- 註解
    https://nttu.shop.tw/products?category=Car'+OR+1=1--
    • SQL 註解為 --
    • +在網址列為 %20 表示空白的意思
  • 註解後的 SQL 語法
    SELECT * FROM products WHERE category = 'Car' OR 1=1--' AND released = 1

影響應用程式邏輯

修改 SQL 語法來影響應用程式的邏輯。

  • 假設一個可以利用帳號密碼登入的系統，輸入帳號密碼後 SQL 語句
    SELECT * FROM users WHERE username = 'admin' AND password = 'admin'
  • 帳號: admin；密碼: 隨便輸入密碼
    SELECT * FROM users WHERE username = 'administrator'--' AND password = '123'

XSS（Cross-Site Scripting, 跨網站腳本）

攻擊者輸入惡意的 HTML 或 JavaScript，讓受害者瀏覽並執行網頁。


圖片來源

XSS 攻擊種類

反射型 XSS

來自當前的 HTTP 請求
因為伺服器而產生的安全風險。

  1. 於請求中夾帶惡意語法或程式碼，可能夾帶在 URL 當中
  2. 常在搜尋框、使用者登入口具有該種弱點
  3. 常用於社交工程手法：釣魚信件，誘使受害者點擊惡意連結，以竊取使用者 Cookie
  4. 範例如下：
    http://www.example.com/upload.asp?id=<script>alert(1);</script>
  • 這種手法要能夠成功攻擊，需要使用社交工程釣魚的技巧，使 User 點擊 URL 攻擊才會生效。
  • 因為 URL 通常看起來很詭異，所以 Hacker 通常會使用短網址或 HTML Encoder 的方式嘗試欺騙 User。

HTML Encoder

  1. 這是為了避免特殊符號造成的顯示問題，以及避免 HTML 將 URL 中的特殊符號視為語法而產生的編碼
  2. 範例如下：
    <script>alert(1);</script> 會被轉換成%3Cscript%3Ealert(1)%3B%3C%2Fscript%3E
  • 但在這裡 Hacker 會反其道而行故意將要輸入在 URL 的特殊符號先轉換成 HTML Encoder 輸入，讓 WebSever 解析成特殊符號，進而進行 XSS 攻擊。

儲存型 XSS

儲存於受害者資料庫等處
因為伺服器而產生的安全風險。

  1. 此手法不需要使用社交工程釣魚的技巧，也能使 User 受到攻擊
  2. 攻擊的方式是 Hacker 將 Javascript 儲存在伺服器的資料庫中，進而引起使 User 遭受攻擊。
  3. 最常見的例子就是將 Javascript 注入留言板，當下一位 User 瀏覽網頁時，網頁會載入留言板的 Javascript 進而使 User 受到攻擊。
  4. 範例如下：
    我是壞人！ <script>alert(1);</script>
  • 然後當 User 瀏覽網頁的時候，就會因為網頁先載入了當下頁面的惡意程式，於是 User 的頁面就會跳出一個 1 的 alert，以此類推， Hacker 在這裡如果輸入讓 User 傳送 cookie 或是其他惡意程式行為，網頁也會完全照做！

DOM 型 XSS

來自瀏覽器中的程式碼非伺服器中的程式碼
因為提取和執行惡意程式都是由 Browser 端完成，屬於前端 JavaScript 的安全漏洞

  1. 這種手法和反射型 XSS 一樣，都需要使用社交工程釣魚的技巧，使 User 點擊 URL 攻擊才會生效。
  2. Hacker 在 URL 輸入 DOM 物件，把物件嵌入網頁程式碼。
  3. 範例：
    <img src=# onerror=”alert(123)”>

CSRF(Cross-site request forgery, 跨網站請求)

攻擊流程

  • 我們已經登入網站後，並取得 Cookie
    Request:/login --> Response:set-cookie:sessionid=2fethkho
  • 駭客透過釣魚網站或釣魚信件誘使受害者點擊瀏覽
  • 瀏覽器預設會把使用者在「目標網站」的 Cookie 連帶送出，因此雖然該請求是在「惡意網站」發出，但「目標網站」收到請求時因為帶有 credentials，所以會誤以為是合法的請求。
  • 受害者送出非經自身意願的請求

SSRF(Server-side request forgery, 伺服器端請求偽造)

攻擊流程

  1. 一台是「中介伺服器端」，另一台是「內部受保護的伺服器端」，其中「內部受保護伺服器端」處於內網，並無對外的接口，需要透過「中介伺服器端」與外網的「客戶端」聯繫，因此，駭客從「客戶端」，做了一個「偽造請求」給伺服器端。
  2. 「偽造請求」的傳送目標是「內部受保護的伺服器端」
    這個請求 (可以是 JSON 或是 XML 等格式) 是被包裝成「中介伺服器端」可接收的格式，但他實際的目的，是要被「內部受保護的伺服器端」(攻擊目標) 給存取，因此它也要同時符合「內部受保護的伺服器端」所能接收的格式。
  3. 「中介伺服器端」本身並不知道這個「偽造請求」會傷害「內部受保護的伺服器端」
    「中介伺服器端」是無意間被當成一個「跳板」，將「偽造請求」送去傷害「內部受保護的伺服器端」，真正做壞事的人是來自客戶端的駭客。

如何做出一個偽造請求？

「內部受保護的伺服器端」很有可能是去呼叫 Google API、Facebook API、AWS API 、Apple API 等等第三方 Open API (如下圖所示)，而這些 API 的介面定義 (包含 endpoint 跟 payload )，都可以在公開的官方文件裡面找到。

要怎麼突破「中介伺服器端」讓它將「偽造請求」往後傳遞？

最常見的方式是透過「目錄遍歷」 (Path Traversal) 攻擊手法
就是客戶端的偽造請求，可以透過「目錄遍歷」 (Path Traversal ) 攻擊手法，指定「中介伺服器端」要呼叫「內部受保護的伺服器端」的哪個 API 網址 ( endpoint )。

目錄遍歷 (Path Traversal)

駭客就是利用這種目錄解析的原理，透過../ 在目錄間穿梭，移動到他不該移動到的目錄底下，存取機密資訊。

Path Traversal 攻擊手法

  • 前端頁面網址的目錄遍歷
    1. 架設網站的時候，會將網頁放在伺服器的某個目錄之下
    2. 舉例來說，你架設了一個網站，網址為 https://game.com 而網頁檔案的目錄如下所示：
      /home/document/123.html
      /home/app.conf
    3. 網址 https://game.com/document/123.html對應到 /home/document/123.html 這個檔案，而 app.conf 是網站的設定檔，裡面有重要的機密設定資料，例如資料庫連線用的帳號密碼等等。
    4. 駭客會利用 ../ 指令企圖去存取 app.conf ，用以下網址去存取你的網站：
      http://game.com/document/../app.conf
    5. 上述該網址對應到伺服器實際的目錄，即為/home/document/../app.conf，此路徑又同義於/home/app.conf
    6. 在真正攻擊的時候通常不能直接使用 ../ 在網址上，因為這個加了../ 是要被傳送到「伺服器」端做解析的，變成無法傳遞到「伺服器」端，而造成攻擊失敗
    7. 若是要避免客戶端解析我們傳送的目錄，便需要對 ../ 進行 Url Encode 變成 ..%2F，透過編碼後的參數 ..%2Fapp.conf 就可以成功對應到 GET /documenmt 這個路由，由伺服器端解析目錄，成功達成攻擊。
