網址和網域組成

一個網址是由通訊協定、主機名稱(或子網域)與網域組成

  • https://www.nttu.edu.tw/ → 網址
  • https → 通訊協定
  • www → 主機名稱，或子網域(大多用來區分網站功能。)
  • nttu.edu.tw → 網域名稱
  • nttu → 次級網域(以學校的英文名稱或代表字來做設定與命名。)
  • edu.tw → 頂級網域(像是網站的分類，政府組機關.edu或台灣網域.tw)

TCP/IP 通訊協定

<h3>目前為網際網路的基礎通訊架構，分為 4 層：</h3>

<img src="https://hackmd.io/_uploads/S16uMryg1g.png" alt="TCP/IP 四層架構">

<ul>
    <li>
        <h4>第四層 - 應用層: 提供了網絡應用程序需要的各種服務。</h4>
        <p><code>HTTP、HTTPS、FTP、SMTP、SSH、DNS …</code></p>
    </li>
    <li>
        <h4>第三層 - 傳輸層: 負責在不同主機之間提供端到端的通信。</h4>
        <p><code>TCP / UDP</code></p>
    </li>
    <li>
        <h4>第二層 - 網路互連層: 負責在不同網絡之間的數據傳輸。</h4>
        <p><code>IP (Internet Protocol)</code></p>
    </li>
    <li>
        <h4>第一層 - 網路存取層: 負責在相鄰網絡節點之間的物理和邏輯鏈路的傳輸。</h4>
        <p><code>乙太網（Ethernet）、Wi-Fi（無線局域網協議）</code></p>
    </li>
</ul>

<br>

<h3>資料傳遞流程</h3>

<img src="https://hackmd.io/_uploads/rk6JwrkgJl.png" alt="資料傳遞流程">

<ol>
    <li>A <code>應用層</code> 將資料傳送給 <code>傳輸層</code>。在 <code>傳輸層</code>，資料被分割成較小的封包</li>
    <li>A <code>傳輸層</code> 使用 TCP 或 UDP 協定來將封包封裝成傳輸層資料段</li>
    <li>A <code>傳輸層</code> 將資料段傳遞給 A <code>網際網路層</code>，並指定目的地的 IP 位址。</li>
    <li>在 A <code>網際網路層</code>，IP 協定負責將資料段封裝成 IP 封包。IP 封包包含源 IP 位址和目的 IP 位址。</li>
    <li>IP 封包通過網際網路傳輸，在路由器之間進行轉發，直到到達目的地網路 B。</li>
    <li>在目的地網路上，目的地 B 主機的 <code>網際網路層</code> 接收到 IP 封包。</li>
    <li>B <code>網際網路層</code> 根據 IP 封包中的目的 IP 位址，將封包傳遞給相應的 B <code>傳輸層</code> 協定（TCP 或 UDP）。</li>
    <li>目的地 B <code>傳輸層</code> 解析接收到的資料段，並將資料段重新組合成原始的資料。</li>
    <li>最後，原始的資料在目的地 B <code>應用層</code> 進行處理和應用。</li>
</ol>

<br>

<h3>TCP(Transmission Control Protocol)</h3>

<blockquote>
    <p>可提供一種可靠的位元組流保證資料完整、無失真並且按順序到達。</p>
</blockquote>

<ol>
    <li>傳輸資料被分成好幾個封包傳送時，當目的端電腦接收到每一個封包，就會傳回一個確認值</li>
    <li>對資料的掌控上及偵錯方面，就遠勝於 UDP，但速度則不如 UDP。</li>
</ol>

<h4>TCP 三次握手 (Three-way Handshake)</h4>

<ol>
    <li>建立連接 (Browser：安安，在嗎？)</li>
    <li>伺服器回應 (Server 收到後回覆：在呀)</li>
    <li>完成握手 (Browser 收到後回覆：收到，太好了)</li>
</ol>
<img src="https://hackmd.io/_uploads/SJi9YrJgJx.png" alt="TCP 三次握手">

<br>

<h3>UDP(User Datagram Protocol)</h3>

<blockquote>
    <p>不檢查封包是否已經到達目的地，並且不保證它們按順序到達。</p>
</blockquote>

<ol>
    <li>只負責送出及接收，不會收到一個封包就回傳一個確認的訊息，檢查則是靠著總和檢測值來判定。</li>
    <li>典型性應用：串流媒體（音訊和視訊等）這樣按時到達比可靠性更重要的應用</li>
</ol>

<h4>UDP 傳輸原理</h4>

<p>傳輸方（Server）不需確認接收方（Client）是否有收到封包</p>
<img src="https://hackmd.io/_uploads/HJUU9r1gJg.png" alt="UDP 傳輸原理">

<br>

<h3>DNS(Domain Name System)</h3>

<blockquote>
    <p>用來統整 IP 與網址對應的系統，像是網際網路的電話簿</p>
</blockquote>

<ol>
    <li>用來做為 IP 與網址的轉換</li>
    <li>透過 DNS Server 的查詢才能讓網站的瀏覽者能正確的查看到他想看的網站</li>
</ol>

<h4>DNS 運作原理</h4>

<p>當你在瀏覽器輸入一串網址，瀏覽器會將這串網址送到離他最近的 DNS Server 去做查詢辨識，當他找到對應的 IP 時，就會回傳這組 IP 的網站資料給你，而若沒有查詢到，則會發生像是 DNS NOT FOUND，網頁或服務無法開啓的情況。</p>
<img src="https://hackmd.io/_uploads/ryMEhBklkl.png" alt="DNS 運作原理">
<img src="https://hackmd.io/_uploads/SyGHhrJxkl.png" alt="DNS 運作原理">

<h3>Http Request Method</h3>

<blockquote>
    <ol>
        <li>發送的 Request 根據不同的用途，有查詢、新增、修改或是刪除，稱為「Request Method」</li>
        <li>是為了讓 Server 能夠清楚辨別 request 的目的</li>
    </ol>
</blockquote>

<h4>HTTP GET</h4>

<ol>
    <li>跟 server 要一個連結或圖片，通常網頁都是 Get 的 request 比較多</li>
    <li>因為只是要獲取資訊、沒有 request body</li>
</ol>

<h4>HTTP POST</h4>

<ol>
    <li>需要執行一些動作時，會傳送 Post 的 request</li>
    <li>獲取「 指定的 」資訊，放在 request body ( Form data ) 裡面</li>
</ol>

<h4>其他的 Http Request Method</h4>

<p>
    <code>Put</code> : 取代掉整個 request<br>
    <code>Patch</code> : 修改部分 request （ 較推薦 ）<br>
    <code>Delete</code> : 刪除資源<br>
    <code>Head</code>: 只要獲取 request 的 header，不要 body<br>
    <code>Option</code> : 可以瞭解 server 提供哪些溝通方法
</p>

<h4>HTTP Code</h4>

<ul>
    <li><code>1xx</code>: 再等等(Server 成功接收、但 Client 還要進行一些處理)</li>
    <li><code>2xx</code>: 成功了
        <ul>
            <li><code>200 OK</code>：連線成功</li>
        </ul>
    </li>
    <li><code>3xx</code>: 去其他地方(重新導向)</li>
    <li><code>4xx</code>: 你挫賽了（Client 端錯誤）
        <ul>
            <li><code>400 Bad Request</code>：請求語法錯誤、或資源太大…等等</li>
            <li><code>401 Unauthorized</code>：未認證，可能需要登入或 Token</li>
            <li><code>403 Forbidden</code>：沒有權限</li>
            <li><code>404 Not Found</code>：找不到資源</li>
        </ul>
    </li>
    <li><code>5xx</code>: 我挫賽了（Server 端錯誤）
        <ul>
            <li><code>500 Internal Server Error</code>：伺服器出錯，搶票時很可能發生</li>
        </ul>
    </li>
</ul>

<h2>SQL injection</h2>

<blockquote>
    <p>利用網頁程式設計者忽略檢查使用者輸入內容造成攻擊</p>
</blockquote>

<br>

<h3>甚麼是 SQL</h3>

<blockquote>
    <p>就是資料庫當中用來查詢資料的語言，所謂資料庫就是存放資料的地方。</p>
</blockquote>

<h3>資料庫的結構</h3>

<ul>
    <li>資料庫名稱(database) &gt; 資料表名稱(table) &gt; 資料欄位名稱(column) &gt; 資料(data)</li>
    <li><code>SELECT * FROM `detail`</code></li>
    <li><img src="https://hackmd.io/_uploads/r1ymu1Ggyg.png" alt="資料庫結構"></li>
</ul>

<br>

<h3>常見的 SQL injection 類型</h3>

<h4>查詢隱藏的數據</h4>

<blockquote>
    <p>修改 SQL 語法來查詢資料庫的其他資料。</p>
</blockquote>

<ul>
    <li>假設一個購物商場透過 URL 對伺服器請求，查詢類別為 <code>Car</code>
        <br><code>https://nttu.shop.tw/products?category=Car</code>
    </li>
    <li>參數 <code>Car</code> 的內容會傳到後端並進行 SQL 語法的查詢
        <br><code>SELECT * FROM products WHERE category = 'Car' AND released = 1</code>
    </li>
    <li>此 SQL 語法的內容為
        <ol>
            <li>選擇所有欄位名稱</li>
            <li>從資料表 products</li>
            <li>category 欄位為 Car</li>
            <li>且 released 發布狀態為 1 （已發布）</li>
        </ol>
    </li>
    <li>我們可以嘗試將 <code>AND released = 1</code> 透過 <code>--</code> 註解
        <br><code>https://nttu.shop.tw/products?category=Car'+OR+1=1--</code>
        <ul>
            <li>SQL 註解為 <code>--</code></li>
            <li><code>+</code> 在網址列為 <code>%20</code> 表示空白的意思</li>
        </ul>
    </li>
    <li>註解後的 SQL 語法
        <br><code>SELECT * FROM products WHERE category = 'Car' OR 1=1--' AND released = 1</code>
    </li>
</ul>

<br>

<h4>影響應用程式邏輯</h4>

<blockquote>
    <p>修改 SQL 語法來影響應用程式的邏輯。</p>
</blockquote>

<ul>
    <li>假設一個可以利用帳號密碼登入的系統，輸入帳號密碼後 SQL 語句
        <br><code>SELECT * FROM users WHERE username = 'admin' AND password = 'admin'</code>
    </li>
    <li>帳號: <code>admin</code>；密碼: <code>隨便輸入密碼</code>
        <br><code>SELECT * FROM users WHERE username = 'administrator'--' AND password = '123'</code>
    </li>
</ul>

<h2>XSS（Cross-Site Scripting, 跨網站腳本）</h2>

<blockquote>
    <p>攻擊者輸入惡意的 HTML 或 JavaScript，讓受害者瀏覽並執行網頁。</p>
</blockquote>

<img src="https://hackmd.io/_uploads/BJ7NQSGe1g.png" alt="XSS 圖片">
<p><a href="https://ithelp.ithome.com.tw/articles/10243967">圖片來源</a></p>

<br>

<h3>XSS 攻擊種類</h3>

<h4>反射型 XSS</h4>

<blockquote>
    <p>來自當前的 HTTP 請求<br>
    因為伺服器而產生的安全風險。</p>
</blockquote>

<ol>
    <li>於請求中夾帶惡意語法或程式碼，可能夾帶在 URL 當中</li>
    <li>常在搜尋框、使用者登入口具有該種弱點</li>
    <li>常用於社交工程手法：釣魚信件，誘使受害者點擊惡意連結，以竊取使用者 Cookie</li>
    <li>範例如下：
        <br><code>http://www.example.com/upload.asp?id=&lt;script&gt;alert(1);&lt;/script&gt;</code>
    </li>
</ol>

<ul>
    <li>這種手法要能夠成功攻擊，需要使用社交工程釣魚的技巧，使 User 點擊 URL 攻擊才會生效。</li>
    <li>因為 URL 通常看起來很詭異，所以 Hacker 通常會使用短網址或 HTML Encoder 的方式嘗試欺騙 User。</li>
</ul>

<h4>HTML Encoder</h4>

<ol>
    <li>這是為了避免特殊符號造成的顯示問題，以及避免 HTML 將 URL 中的特殊符號視為語法而產生的編碼</li>
    <li>範例如下：
        <br><code>&lt;script&gt;alert(1);&lt;/script&gt;</code> 會被轉換成 <code>%3Cscript%3Ealert(1)%3B%3C%2Fscript%3E</code>
    </li>
</ol>

<ul>
    <li>但在這裡 Hacker 會反其道而行故意將要輸入在 URL 的特殊符號先轉換成 HTML Encoder 輸入，讓 WebSever 解析成特殊符號，進而進行 XSS 攻擊。</li>
</ul>

<br>

<h4>儲存型 XSS</h4>

<blockquote>
    <p>儲存於受害者資料庫等處<br>
    因為伺服器而產生的安全風險。</p>
</blockquote>

<ol>
    <li>此手法不需要使用社交工程釣魚的技巧，也能使 User 受到攻擊</li>
    <li>攻擊的方式是 Hacker 將 Javascript 儲存在伺服器的資料庫中，進而引起使 User 遭受攻擊。</li>
    <li>最常見的例子就是將 Javascript 注入留言板，當下一位 User 瀏覽網頁時，網頁會載入留言板的 Javascript 進而使 User 受到攻擊。</li>
    <li>範例如下：
        <br><code>我是壞人！ &lt;script&gt;alert(1);&lt;/script&gt;</code>
    </li>
</ol>

<ul>
    <li>然後當 User 瀏覽網頁的時候，就會因為網頁先載入了當下頁面的惡意程式，於是 User 的頁面就會跳出一個 1 的 alert，以此類推， Hacker 在這裡如果輸入讓 User 傳送 cookie 或是其他惡意程式行為，網頁也會完全照做！</li>
</ul>

<br>

<h4>DOM 型 XSS</h4>

<blockquote>
    <p>來自瀏覽器中的程式碼非伺服器中的程式碼<br>
    因為提取和執行惡意程式都是由 Browser 端完成，屬於前端 JavaScript 的安全漏洞</p>
</blockquote>

<ol>
    <li>這種手法和反射型 XSS 一樣，都需要使用社交工程釣魚的技巧，使 User 點擊 URL 攻擊才會生效。</li>
    <li>Hacker 在 URL 輸入 DOM 物件，把物件嵌入網頁程式碼。</li>
    <li>範例：
        <br><code>&lt;img src=# onerror=”alert(123)”&gt;</code>
    </li>
</ol>

<h4>DOM(Document Object Model)</h4>

<blockquote>
    <p>是用來描述 HTML 文件的表示法，可以使用 JavaScript 來動態產生完整的網頁，而不必透過伺服器。</p>
</blockquote>

<ol>
    <li>簡單來說就是把一份 HTML 文件中的各種標籤及文字、圖片等，都定義成一個個網頁物件，而這些物件最終會成為一個樹狀結構</li>
    <li>範例如下：
        <br><img src="https://hackmd.io/_uploads/ryLarBzlJg.png" alt="DOM 結構">
    </li>
</ol>

<h2>CSRF(Cross-site request forgery, 跨網站請求)</h2>

<h3>攻擊流程</h3>

<img src="https://hackmd.io/_uploads/SJ00DHfeJg.png" alt="CSRF 攻擊流程">

<ul>
    <li>我們已經登入網站後，並取得 Cookie
        <br><code>Request:/login --&gt; Response:set-cookie:sessionid=2fethkho</code>
    </li>
    <li>駭客透過釣魚網站或釣魚信件誘使受害者點擊瀏覽</li>
    <li>瀏覽器預設會把使用者在「目標網站」的 Cookie 連帶送出，因此雖然該請求是在「惡意網站」發出，但「目標網站」收到請求時因為帶有 credentials，所以會誤以為是合法的請求。</li>
    <li>受害者送出非經自身意願的請求</li>
</ul>

<h2>SSRF(Server-side request forgery, 伺服器端請求偽造)</h2>

<h3>攻擊流程</h3>

<img src="https://hackmd.io/_uploads/SJ5JpSGlke.png" alt="SSRF 攻擊流程">

<ol>
    <li>一台是「中介伺服器端」，另一台是「內部受保護的伺服器端」，其中「內部受保護伺服器端」處於內網，並無對外的接口，需要透過「中介伺服器端」與外網的「客戶端」聯繫，因此，駭客從「客戶端」，做了一個「偽造請求」給伺服器端。</li>
    <li>「偽造請求」的傳送目標是「內部受保護的伺服器端」
        <br>這個請求 (可以是 JSON 或是 XML 等格式) 是被包裝成「中介伺服器端」可接收的格式，但他實際的目的，是要被「內部受保護的伺服器端」(攻擊目標) 給存取，因此它也要同時符合「內部受保護的伺服器端」所能接收的格式。
    </li>
    <li>「中介伺服器端」本身並不知道這個「偽造請求」會傷害「內部受保護的伺服器端」
        <br>「中介伺服器端」是無意間被當成一個「跳板」，將「偽造請求」送去傷害「內部受保護的伺服器端」，真正做壞事的人是來自客戶端的駭客。
    </li>
</ol>

<br>

<h3>如何做出一個偽造請求？</h3>

<blockquote>
    <p>「內部受保護的伺服器端」很有可能是去呼叫 Google API、Facebook API、AWS API 、Apple API 等等第三方 Open API (如下圖所示)，而這些 API 的介面定義 (包含 endpoint 跟 payload )，都可以在公開的官方文件裡面找到。</p>
    <img src="https://hackmd.io/_uploads/SkpyeIzgkx.png" alt="第三方 API">
</blockquote>

<h3>要怎麼突破「中介伺服器端」讓它將「偽造請求」往後傳遞？</h3>

<blockquote>
    <p>最常見的方式是透過「目錄遍歷」 (Path Traversal) 攻擊手法<br>
    就是客戶端的偽造請求，可以透過「目錄遍歷」 (Path Traversal ) 攻擊手法，指定「中介伺服器端」要呼叫「內部受保護的伺服器端」的哪個 API 網址 ( endpoint )。</p>
</blockquote>

<h3>目錄遍歷 (Path Traversal)</h3>

<blockquote>
    <p>駭客就是利用這種目錄解析的原理，透過 <code>../</code> 在目錄間穿梭，移動到他不該移動到的目錄底下，存取機密資訊。</p>
</blockquote>

<h4>Path Traversal 攻擊手法</h4>

<ul>
    <li>前端頁面網址的目錄遍歷
        <ol>
            <li>架設網站的時候，會將網頁放在伺服器的某個目錄之下</li>
            <li>舉例來說，你架設了一個網站，網址為 <code>https://game.com</code> 而網頁檔案的目錄如下所示：
                <ul>
                    <li><code>/home/document/123.html</code></li>
                    <li><code>/home/app.conf</code></li>
                </ul>
            </li>
            <li>網址 <code>https://game.com/document/123.html</code> 對應到 <code>/home/document/123.html</code> 這個檔案，而 <code>app.conf</code> 是網站的設定檔，裡面有重要的機密設定資料，例如資料庫連線用的帳號密碼等等。</li>
            <li>駭客會利用 <code>../</code> 指令企圖去存取 <code>app.conf</code>，用以下網址去存取你的網站：
                <br><code>http://game.com/document/../app.conf</code>
            </li>
            <li>上述該網址對應到伺服器實際的目錄，即為 <code>/home/document/../app.conf</code>，此路徑又同義於 <code>/home/app.conf</code></li>
            <li>在真正攻擊的時候通常不能直接使用 <code>../</code> 在網址上，因為這個加了 <code>../</code> 是要被傳送到「伺服器」端做解析的，變成無法傳遞到「伺服器」端，而造成攻擊失敗
                <br><img src="https://hackmd.io/_uploads/SylwIUzgJl.png" alt="目錄遍歷失敗">
            </li>
            <li>若是要避免客戶端解析我們傳送的目錄，便需要對 <code>../</code> 進行 Url Encode 變成 <code>..%2F</code>，透過編碼後的參數 <code>..%2Fapp.conf</code> 就可以成功對應到 GET /document 這個路由，由伺服器端解析目錄，成功達成攻擊。
                <br><img src="https://hackmd.io/_uploads/SkGuUUzxke.png" alt="目錄遍歷成功">
            </li>
        </ol>
    </li>
</ul>
